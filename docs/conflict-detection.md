# Martenweave — Conflict Detection for Concurrent Model Changes

Version: `0.1-draft`  
Document type: Architecture / safety design  
Scope: How to prevent accidental overwrites when multiple changes target the same model objects  
Depends on: `docs/integration-architecture.md`, `docs/bulk-refactor-operations.md`  
Status: Draft for implementation

---

## 1. Purpose

This document defines **conflict detection** for Martenweave. When multiple proposals, spreadsheet imports, source refreshes, or Git branches change the same model objects, the system must detect the overlap and prevent silent overwrites of canonical model truth.

Conflict detection is **advisory**, not automatic. The system warns users about conflicts; it does not resolve them without human decision.

---

## 2. Design principles

### 2.1 Never silently overwrite canonical truth

- If two changes target the same object, at least one must be reviewed before either is applied.
- The canonical file on disk is the ground truth for conflict detection.
- A PatchProposal is "stale" if the canonical file has changed since the proposal was generated.

### 2.2 Proposal-first for all conflict resolution

- Detected conflicts produce a warning or a new "conflict resolution" PatchProposal.
- Humans decide which changes to keep, merge, or discard.
- No automatic merge or three-way resolution in MVP.

### 2.3 Explicit checks only

- Users must run a command to check for conflicts: `modelops proposal conflicts`.
- No background polling or automatic conflict detection.
- CI may run conflict checks as part of validation.

## 3. Conflict types

### 3.1 Same object changed by multiple proposals

```text
Proposal A: update ATTR-TEST.name = "Name A"
Proposal B: update ATTR-TEST.name = "Name B"
```

**Detection:** Compare `affected_objects` across open PatchProposals.

**Resolution:** Human reviews both proposals and approves one (or creates a merged proposal).

### 3.2 Stale spreadsheet edit

```text
User exports model to Excel
  -> User edits ATTR-TEST.name = "New Name"
  -> Another user edits canonical file directly: ATTR-TEST.name = "Direct Edit"
  -> First user re-imports spreadsheet
```

**Detection:** The re-imported spreadsheet differs from the current canonical file in ways the proposal did not anticipate. The `source_evidence` timestamp or checksum in the proposal does not match the current canonical state.

**Resolution:** Import produces a PatchProposal with a "stale_source" warning. Human reviews the diff.

### 3.3 Source refresh conflicts with manual edit

```text
Dataset profile D1 references ATTR-TEST
  -> Source refresh detects dataset has changed
  -> User has manually edited ATTR-TEST since last profile
```

**Detection:** The refreshed profile suggests changes to an object whose canonical file has a newer `modified_at` than the last source registry entry.

**Resolution:** Refresh produces a proposal with a "manual_edit_conflict" warning.

### 3.4 Deleted object referenced by proposal

```text
Proposal A: create MAP-1 linking ATTR-OLD -> FEP-1
  -> ATTR-OLD is deleted by another approved ChangeRequest
```

**Detection:** Validation of Proposal A fails because `ATTR-OLD` no longer exists.

**Resolution:** Proposal validation fails with a broken reference error. Human must update the proposal.

### 3.5 Relationship / LoV / rule conflicts

```text
Proposal A: attach ValueList VL-1 to ATTR-TEST
Proposal B: attach ValueList VL-2 to ATTR-TEST
```

**Detection:** Both proposals modify the same relationship field.

**Resolution:** Human reviews both proposals. Only one ValueList can be attached (unless the field supports multiple).

## 4. Conflict metadata

### 4.1 Per-object checksum

Each canonical object can carry a lightweight checksum of its frontmatter:

```yaml
---
id: ATTR-TEST
type: Attribute
status: draft
name: Test Attribute
_checksum: "sha256:abc123..."
---
```

The `_checksum` is computed from all frontmatter fields except `_checksum` itself. It is regenerated by `modelops build-index` or `modelops validate`.

### 4.2 Per-proposal snapshot

When a PatchProposal is generated, it stores the current checksums of all affected objects:

```json
{
  "id": "PP-IMPORT-001",
  "affected_objects": ["ATTR-TEST", "FEP-1"],
  "object_checksums": {
    "ATTR-TEST": "sha256:abc123...",
    "FEP-1": "sha256:def456..."
  }
}
```

### 4.3 Stale proposal detection

A proposal is stale if any affected object's current checksum differs from the snapshot:

```python
def is_proposal_stale(proposal, current_checksums):
    for obj_id, snapshot_checksum in proposal.get("object_checksums", {}).items():
        if current_checksums.get(obj_id) != snapshot_checksum:
            return True, obj_id
    return False, None
```

## 5. CLI design (future)

### 5.1 Check conflicts for a proposal

```bash
modelops proposal conflicts <proposal-id> --repo ./my-model
```

**Output:**

```text
Conflict check for PP-IMPORT-001:
  Stale objects: 1
    - ATTR-TEST: checksum mismatch (proposal: abc123..., current: def456...)
  Overlapping proposals: 1
    - PP-AI-002 also affects ATTR-TEST
  Broken references: 0

Recommendation: Review stale objects before applying.
```

### 5.2 Check all open proposals

```bash
modelops proposal conflicts --all --repo ./my-model
```

### 5.3 Apply with conflict guard

```bash
modelops proposal apply <proposal-id> --repo ./my-model --apply
```

**Behavior:**

1. Run conflict check internally.
2. If stale objects detected, block apply and print conflict report.
3. If `--force` is passed, warn strongly and require confirmation.
4. If no conflicts, proceed with apply.

## 6. Conflict resolution strategies

### 6.1 Re-generate proposal

```bash
modelops propose-patch --from ./note.md --repo ./my-model
```

The new proposal uses current canonical state as its baseline, eliminating staleness.

### 6.2 Merge proposals (future)

```bash
modelops proposal merge PP-IMPORT-001 PP-AI-002 --repo ./my-model
```

Produces a new PatchProposal with non-conflicting operations from both sources, and warnings for conflicting fields.

### 6.3 Manual resolution

User edits canonical files directly, then re-runs validation. The stale proposal is abandoned.

## 7. Integration with existing flows

| Flow | Conflict check | Action on conflict |
|---|---|---|
| Spreadsheet import | Compare checksums at import time | Add `stale_source` warning to proposal |
| Source refresh | Compare source registry timestamp with file mtime | Add `manual_edit_conflict` warning |
| AI patch proposal | Store checksums in proposal | Block apply if stale |
| Git bundle | Not applicable (bundle is offline) | User resolves at merge time |
| Bulk refactor | Check overlapping proposals before apply | Block if overlap detected |

## 8. Implementation notes

### 8.1 Checksum computation

```python
import hashlib
import json

def compute_object_checksum(frontmatter: dict) -> str:
    """Compute a stable checksum for canonical object frontmatter."""
    data = {k: v for k, v in frontmatter.items() if k != "_checksum"}
    canonical = json.dumps(data, sort_keys=True, default=str)
    return hashlib.sha256(canonical.encode("utf-8")).hexdigest()[:16]
```

### 8.2 Proposal staleness check

```python
def check_proposal_stale(proposal: dict, repo_model_path: Path) -> list[str]:
    """Return warnings if proposal is stale relative to current canonical files."""
    from modelops_core.imports.model_sheet_import_service import _load_existing_objects

    warnings: list[str] = []
    current = _load_existing_objects(repo_model_path)
    snapshot = proposal.get("object_checksums", {})

    for obj_id, expected_checksum in snapshot.items():
        if obj_id not in current:
            warnings.append(f"Object {obj_id} was deleted since proposal was created.")
            continue
        actual_checksum = compute_object_checksum(current[obj_id])
        if actual_checksum != expected_checksum:
            warnings.append(
                f"Object {obj_id} has changed since proposal was created."
            )

    return warnings
```

### 8.3 Overlap detection

```python
def detect_overlapping_proposals(
    proposals: list[dict],
) -> dict[str, list[str]]:
    """Return a map of object_id -> list of proposal_ids that affect it."""
    overlaps: dict[str, list[str]] = {}
    for proposal in proposals:
        pid = proposal.get("id", "unknown")
        for obj_id in proposal.get("affected_objects", []):
            overlaps.setdefault(obj_id, []).append(pid)
    return {k: v for k, v in overlaps.items() if len(v) > 1}
```
