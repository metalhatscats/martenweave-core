# dbt and Analytics Model Import Design

## Goal
Design how Martenweave can import dbt and analytics model metadata to connect analytical models with business attributes, source systems, validation rules, and ownership.

## Scope

This document covers:

- Mapping from dbt project artifacts to Martenweave objects
- Mapping from dbt tests to validation and data quality objects
- Mapping from dbt lineage to Martenweave relationship structures
- Safe import flow through PatchProposal
- Future CLI command design

## dbt Concept → Martenweave Concept Map

| dbt Concept | Martenweave Object | Notes |
|---|---|---|
| `source` | `Dataset` | External data source consumed by dbt models |
| `model` | `BusinessEntity` or `Dataset` | Semantic model → Entity; staging table → Dataset |
| `seed` | `Dataset` | Static CSV data loaded as a table |
| `snapshot` | `Dataset` | Slowly-changing dimension snapshot |
| `exposure` | `BusinessEntity` | Business-facing output (dashboard, API, report) |
| `column` | `Attribute` + `FieldEndpoint` | Column description → Attribute; column name/type → FieldEndpoint |
| `description` | `Attribute.description` or `Entity.description` | dbt descriptions carried forward |
| `tags` | `Attribute` metadata or `Domain` | Tags can group objects into domains |
| `meta` | `Attribute` metadata or custom facets | Arbitrary key-value metadata |
| `test: unique` | `ValidationRule` | Uniqueness constraint |
| `test: not_null` | `ValidationRule` | Required field constraint |
| `test: accepted_values` | `ValueList` + `ValidationRule` | Enum values become ValueList; validation ensures match |
| `test: relationships` | `Mapping` | Foreign-key relationship between models |
| `test: custom` | `DataQualityCheck` or `ValidationRule` | Custom test logic mapped to data quality check |
| `ref()` | `Mapping` or lineage edge | Model dependency expressed as mapping or relationship |
| `source()` | `Mapping` or lineage edge | Source-to-model dependency |
| `dbt_project.yml` | `System` | The dbt project as a system in the landscape |
| `profile` target | `SystemEnvironment` | Dev/staging/prod environments |

## Import Flow

```
1. Parse dbt artifacts
   └─ Read manifest.json (generated by dbt docs generate)
   └─ Read catalog.json (optional, for column types and stats)
   └─ Do not parse SQL files directly

2. Create SourceEntry
   └─ Register the dbt project as a source
   └─ Store manifest path and version in SourceRegistryService

3. Generate candidates
   └─ Map sources to Dataset candidates
   └─ Map models/seeds/snapshots to Entity or Dataset candidates
   └─ Map exposures to Entity candidates
   └─ Map columns to Attribute + FieldEndpoint candidates
   └─ Map tests to ValidationRule or DataQualityCheck candidates
   └─ Map refs and sources to Mapping candidates
   └─ Map tags to Domain candidates or metadata

4. Build PatchProposal
   └─ Package candidates as PatchProposal
   └─ Proposal ID: PP-IMPORT-DBT:<manifest_hash>
   └─ Include metadata: dbt project name, version, import timestamp

5. Validate and review
   └─ Run through standard validation pipeline
   └─ Human reviews and edits generated objects
   └─ Approve → ChangeRequest → Apply
```

## Example: Importing a dbt Model

Input dbt model (`models/marts/customers.sql`):

```sql
{{ config(materialized='table') }}

select
    customer_id,
    first_name,
    last_name,
    email,
    created_at
from {{ ref('stg_customers') }}
```

Input dbt schema (`models/marts/customers.yml`):

```yaml
version: 2

models:
  - name: customers
    description: "Customer dimension table"
    columns:
      - name: customer_id
        description: "Unique customer identifier"
        tests:
          - unique
          - not_null
      - name: email
        description: "Customer email address"
        tests:
          - not_null
```

Generated Martenweave candidates:

```yaml

status: active
name: Analytics dbt Project

# Dataset for source

# Entity for model

type: BusinessEntity
status: draft

# Attribute + FieldEndpoint for customer_id
```

## Test Mapping Details

### Built-in dbt Tests

| dbt Test | Martenweave Object | Rule Type |
|---|---|---|
| `unique` | `ValidationRule` | `unique` |
| `not_null` | `ValidationRule` | `not_null` |
| `accepted_values` | `ValidationRule` + `ValueList` | `accepted_values` |
| `relationships` | `Mapping` | — |
| `dbt_utils.equal_rowcount` | `DataQualityCheck` | `equal_rowcount` |
| `dbt_utils.recency` | `DataQualityCheck` | `recency` |

### Custom Generic Tests

Custom generic tests (defined in `macros/test_*.sql`) are mapped to `DataQualityCheck` with:
- `rule_type: custom`
- `description`: the test macro name and arguments
- The test SQL is stored as Evidence or metadata, not as a canonical object

## Lineage from dbt

dbt's `ref()` and `source()` dependencies create lineage edges. The import generates:

- `mapped_from` edges: model → source
- `mapped_to` edges: source → model
- `part_of_mapping_set` edges: group related mappings by dbt model

For complex dbt DAGs with many models, the import can optionally create `IntegrationFlow` and `DataFlowStep` objects to represent the full transformation pipeline.

## Connector Design

A dbt connector implements `ConnectorAdapter`:

```python
class DbtConnector:
    @property
    def connector_type(self) -> str:
        return "dbt"

    def list_sources(self, prefix: str = "") -> list[ConnectorSourceInfo]:
        """List models/sources matching prefix from manifest."""
        ...

    def fetch_metadata(self, source_id: str) -> ConnectorSourceInfo:
        """Fetch model metadata (description, columns, tests)."""
        ...

    def fetch_content(self, source_id: str) -> bytes:
        """Return raw SQL file content."""
        ...
```

The connector reads `manifest.json` and optionally `catalog.json` from the dbt project's `target/` directory. No database connection is required for metadata import.

## Future CLI Command

```bash
# Import from a dbt project
modelops import-dbt --repo ./my-model --from ./dbt-project/target/manifest.json

# Import with catalog for column types
modelops import-dbt --repo ./my-model --from ./dbt-project/target/manifest.json \
  --catalog ./dbt-project/target/catalog.json

# Import only specific models
modelops import-dbt --repo ./my-model --from ./manifest.json --models "marts.*"

# Dry-run preview
modelops import-dbt --repo ./my-model --from ./manifest.json --dry-run
```

## Out of Scope

- **SQL parsing** — extracting lineage from raw SQL is not covered. dbt's manifest already contains the dependency graph.
- **dbt Cloud API** — importing from dbt Cloud's REST API is a future enhancement.
- **Runtime execution** — dbt run results (success/failure, row counts) are not imported.
- **Write-back to dbt** — Martenweave does not generate or modify dbt model files.
- **Other analytics tools** — Looker, Tableau, Power BI are not covered. The same pattern could be extended.

## Safety Notes

- The connector only reads JSON artifacts. No database queries are executed.
- SQL content from `fetch_content` is treated as opaque text, not parsed or executed.
- Large dbt projects (>1,000 models) should be filtered by path or tag to avoid overwhelming the proposal builder.
